# Sparkle package-update architecture for Vox

## Decision

Vox should make `VoxHelper.app` the sole Sparkle host and publish **flat, signed, notarized `.pkg` files** through one HTTPS Sparkle 2 appcast. Stable releases stay on Sparkle's default channel; opt-in prereleases use a `beta` channel. The first Sparkle-enabled version is a bridge package that existing users install through the current/manual path. Every later supported release can update through the helper's native Sparkle UI. `scripts/update.sh` remains a documented recovery and source-install mechanism, but it must not compete with Sparkle as the normal update path.

This is the right package shape for Vox because a release updates two app bundles plus `/Library/Application Support/Vox/Bootstrap`, UI/server files, package scripts, and user LaunchAgents. Sparkle recommends regular app-bundle updates when possible, but explicitly supports flat bare packages for custom installations. Its package-update tradeoffs must be accepted: each install needs authorization, silent installation is impossible, relaunch is slower, deltas are unavailable, package feeds are not conveniently generated by `generate_appcast`, and installer-package updates have no signing-key-rotation fallback. [Sparkle package-update documentation](https://sparkle-project.org/documentation/package-updates/)

## Native integration and reproducible dependency build

1. Add a Swift Package Manager manifest for the helper target, depend on `https://github.com/sparkle-project/Sparkle`, and pin an exact production Sparkle tag in `Package.resolved` rather than accepting an unconstrained update. At the time of this research the current documented API reference is 2.9.4, and 2.9.4 contains security/reliability fixes, so that is the appropriate initial pin. Review and deliberately bump it as a dependency update. Sparkle's package itself exposes `Sparkle` as a binary target whose URL and checksum are versioned, which makes the downloaded framework reproducible. [Sparkle 2.9.4 source tag](https://github.com/sparkle-project/Sparkle/tree/2.9.4), [Sparkle security and reliability history](https://sparkle-project.org/documentation/security-and-reliability/)
2. Replace the direct `swiftc` helper compilation in `scripts/build-apps.sh` with `swift build --configuration release --arch arm64` for the helper package. Copy the resolved `Sparkle.framework` into `VoxHelper.app/Contents/Frameworks` with a symlink-preserving tool (`ditto`), and verify the helper executable resolves it from `@executable_path/../Frameworks` (or the equivalent loader rpath) with `otool -L` and `otool -l`. Sparkle's manual/external-build guidance requires the framework search path, runtime search path, framework copy, preserved symlinks, bundle version keys, feed/key keys, deployment target, and architectures. [Sparkle programmatic setup](https://sparkle-project.org/documentation/programmatic-setup/)
3. Keep Vox Helper non-sandboxed unless Vox separately chooses to sandbox it. Therefore do **not** enable `SUEnableInstallerLauncherService` or `SUEnableDownloaderService`; Sparkle says those XPC switches are for sandboxed applications. Even for a non-sandboxed host, the bundled updater tools must be signed correctly for distribution. The current blanket `codesign --deep` signing should be replaced by Sparkle's documented inside-out signing order for nested XPC services/tools, then `Sparkle.framework`, then `VoxHelper.app`; Sparkle specifically warns against using `--deep` in custom signing scripts. [Sparkle sandboxing and manual code-signing guidance](https://sparkle-project.org/documentation/sandboxing/)
4. Retain the current `arm64-apple-macos13.0` deployment target and align `VoxServer.app`'s inconsistent `LSMinimumSystemVersion` (`12.0` today) to the actual supported minimum. Preserve framework symlinks all the way through DMG staging and package payload creation because flattening them breaks the framework signature. [Sparkle basic setup](https://sparkle-project.org/documentation/)

In Swift, `AppDelegate` should own a long-lived `SPUStandardUpdaterController(startingUpdater: true, updaterDelegate: ..., userDriverDelegate: ...)`. Pass it into `StatusBarController`; change “Check for Updates…” to call `checkForUpdates(_:)` and enable the item from `updater.canCheckForUpdates`. This is Sparkle's standard programmatic host for updating the main application bundle and presenting native update UI. [SPUStandardUpdaterController reference](https://sparkle-project.org/documentation/api-reference/Classes/SPUStandardUpdaterController.html), [programmatic API expectations](https://sparkle-project.org/documentation/programmatic-setup/)

The updater delegate should implement only the policy Vox actually needs:

- `allowedChannels(for:)` returns `[]` normally and `{"beta"}` when the user opts into prereleases. The default channel is always included by Sparkle.
- `updater(_:didAbortWithError:)` and/or `updater(_:didFinishUpdateCycleFor:error:)` records a concise error and exposes “Open update log” and “Repair manually” actions. Treat `SUInstallationCanceledError` as cancellation, not corruption.
- Relaunch callbacks should stop showing transient “server restarting” state and let the package scripts/LaunchAgents own server restart. Do not postpone relaunch indefinitely.

Sparkle documents these lifecycle and error callbacks, including the distinct authorization-cancel error. [SPUUpdaterDelegate reference](https://sparkle-project.org/documentation/api-reference/Protocols/SPUUpdaterDelegate.html)

## Required `VoxHelper.app` `Info.plist` contract

Add these keys to the generated helper plist:

```xml
<key>SUFeedURL</key>
<string>https://UPDATE_HOST/vox/appcast.xml</string>
<key>SUPublicEDKey</key>
<string>BASE64_PUBLIC_KEY</string>
<key>SUEnableAutomaticChecks</key>
<true/>
<key>SUAutomaticallyUpdate</key>
<false/>
```

Keep and validate:

```xml
<key>CFBundleIdentifier</key>
<string>com.noelmom.vox-helper</string>
<key>CFBundleVersion</key>
<string>MONOTONIC_NUMERIC_BUILD</string>
<key>CFBundleShortVersionString</key>
<string>HUMAN_SEMVER</string>
<key>LSMinimumSystemVersion</key>
<string>13.0</string>
```

`CFBundleVersion` is the update ordering key and must be monotonically increasing independently of display version. The existing UTC minute build number is only safe if release automation rejects reuse/regression; a persisted CI/release sequence is stronger. Use `CFBundleShortVersionString` for values such as `0.6.0-beta.1`. Sparkle recommends `SUFeedURL` and `SUPublicEDKey` in the plist and defines automatic checking as a default that users may later change. `SUAutomaticallyUpdate` must remain false because package updates always need authorization and cannot be silently installed. [Sparkle distribution requirements](https://sparkle-project.org/documentation/), [Sparkle customization keys](https://sparkle-project.org/documentation/customization/)

Do not initially enable `SURequireSignedFeed`. The package itself is still EdDSA-signed and served over HTTPS, while a signed feed adds operational fragility and Sparkle only began validating signed feeds in 2.9. More importantly, package updates have no signing-key-rotation fallback. Once the bridge has broad adoption, offline key restore has been drill-tested, and every supported client is pinned to Sparkle 2.9+, signed feeds can be a second controlled migration; enabling them also requires `SUVerifyUpdateBeforeExtraction` and re-signing every appcast/release-note modification. [Sparkle feed-signing and key-rotation guidance](https://sparkle-project.org/documentation/)

## EdDSA key and appcast publication

Generate one Ed25519 keypair with Sparkle's `generate_keys`. Store the signing key in the release operator's Keychain (or import it transiently from an encrypted CI secret), export an encrypted offline recovery copy using the tool's supported export option, and keep that copy separate from the update host. Commit only the base64 public key. Sparkle explicitly recommends keeping the private key away from the web host and documents Keychain storage plus export/import. [Sparkle EdDSA setup](https://sparkle-project.org/documentation/)

Because `generate_appcast` does not conveniently generate bare-package appcasts, release automation should maintain a deterministic appcast template and call Sparkle's `sign_update` on the final notarized `.pkg`. The tool returns exactly the `sparkle:edSignature` and byte `length` attributes needed for the enclosure. Do not pass private key material on the command line; Sparkle deprecated that pattern as insecure. [Sparkle publishing guide](https://sparkle-project.org/documentation/publishing/), [Sparkle upgrade notes](https://sparkle-project.org/documentation/upgrading/)

Each item should be structurally equivalent to:

```xml
<item>
  <title>Vox 0.6.0</title>
  <sparkle:version>2026071001</sparkle:version>
  <sparkle:shortVersionString>0.6.0</sparkle:shortVersionString>
  <pubDate>Fri, 10 Jul 2026 12:00:00 +0000</pubDate>
  <description sparkle:format="markdown"><![CDATA[...]]></description>
  <enclosure
    url="https://UPDATE_HOST/vox/releases/Vox-0.6.0.pkg"
    length="..."
    type="application/octet-stream"
    sparkle:edSignature="..." />
</item>
```

For a beta, add `<sparkle:channel>beta</sparkle:channel>`. Do **not** add `sparkle:installationType="package"` to a bare `.pkg`; Sparkle requires that attribute for an archived package and says it is unnecessary for the recommended bare-package path. Use one appcast so beta users see beta plus stable releases and stable users see only the default channel. Channels are for prereleases that ultimately rejoin the default channel, not permanent parallel editions. [Sparkle package updates](https://sparkle-project.org/documentation/package-updates/), [Sparkle channels](https://sparkle-project.org/documentation/publishing/#channels)

For stable releases, add `sparkle:phasedRolloutInterval` only after the bridge and at least one full production update have passed. A 24-hour interval yields Sparkle's fixed seven-group, seven-day rollout. Manual checks and critical updates bypass phasing, so phasing limits automatic exposure rather than acting as an emergency stop. Beta releases should not be phased. [Sparkle phased rollouts](https://sparkle-project.org/documentation/publishing/#phased-group-rollouts)

Host the appcast, release notes, and packages on HTTPS with immutable versioned package URLs. Publish artifacts first and appcast last; an appcast must never reference a package that is not yet globally downloadable. A rollback is a **new, higher** `CFBundleVersion` containing the prior known-good code, never a rewritten package or a lowered version.

## Bridge migration

The bridge is a normal signed/notarized Vox package containing the Sparkle-enabled helper, new feed URL/public key, native menu action, and the still-working shell recovery updater. Existing installations have no Sparkle runtime, so they cannot consume a Sparkle appcast: they must install this bridge once through the current package/manual update route. After bridge installation, all package updates are bare Sparkle updates.

Migration sequence:

1. Ship and retain a manually downloadable bridge `.pkg`; advertise it as the one-time updater migration.
2. Existing git/source users may use the current shell updater to obtain bridge source, but the acceptance test must still install the notarized bridge package because Sparkle warns package updates can fail in development builds whose helper tools have not been re-signed. [Sparkle package testing warning](https://sparkle-project.org/documentation/package-updates/)
3. Point the bridge at a new Sparkle-only appcast from day one. This leaves the legacy update mechanism isolated and allows channels/new feed behavior without compatibility concerns; Sparkle recommends switching feed URLs when newer feed features cannot be assumed across the installed base. [Sparkle upgrade migration guidance](https://sparkle-project.org/documentation/publishing/#upgrading-to-newer-features)
4. Keep the bridge package and last known-good full package downloadable indefinitely. Never require Sparkle merely to repair Sparkle.

## Installer behavior, authorization, and recovery

The current package preinstall performs three live internet checks and postinstall may wait three minutes for the server, re-run bootstrap, and open a welcome page. Those are first-install behaviors and are inappropriate for every background update. Split package scripts by an explicit installed marker/receipt:

- **Fresh install:** prerequisites, dependency bootstrap, LaunchAgent creation, and Welcome launch are allowed.
- **Update:** no arbitrary internet preflight (the package is already downloaded); stop the Vox server cleanly; preserve user `.env`, voices, recordings, outputs, input, database, and settings; replace signed application/bootstrap payload; run only necessary dependency migrations; reinstall/reload agents idempotently; write installed-version state; and exit promptly. Never open Welcome during an update.
- Write an update transaction record before mutation and a completion record only after health validation. Package scripts must be safe to rerun after interruption.

Failure policy by stage:

- **Feed/download/signature failure:** no installed files have changed. Sparkle retains the current app and reports the error; show retry, diagnostics, and the versioned manual package link.
- **Authorization cancellation:** leave the current installation running and report “Update canceled.” Package installs always require authorization, so there is no silent fallback. [Sparkle package limitations](https://sparkle-project.org/documentation/package-updates/)
- **Installer/package-script failure:** never claim success. Preserve `/Library/Logs/Vox/pkg-install.log`, Sparkle/Console diagnostics, the failed target version, and the last known-good package URL. Offer “Repair Vox…” which opens the signed manual package or invokes the recovery script with explicit confirmation. Because package updates have no delta fallback and package scripts can mutate system state, recovery must rely on idempotent scripts plus a complete known-good package, not an assumed automatic rollback.
- **Relaunch/server-health failure after a successful package receipt:** relaunch the helper independently of server health, show a degraded state, and offer restart server, open logs, repair, and reinstall last known-good/newer package. Never loop an automatic reinstall.
- **Lost EdDSA key:** stop publication. Restore the offline key. Do not attempt package key rotation through appcast metadata; Sparkle explicitly says package updates lack that fallback. If recovery is impossible, distribute a new manually installed bridge with a new public key/feed.

`scripts/update.sh` should remain tested and shipped under a clearly named **Recovery / source update** command. For a package install it must never overwrite a Developer-ID-signed app with an ad-hoc development build or `git pull` an unexpected branch. Require an explicit source directory or trusted release checkout, preserve user data, verify expected version/source, and support `--no-restart` for diagnosis. The normal menu item must use Sparkle; the recovery action belongs under Diagnostics/Option-click and must explain that it opens Terminal.

## Release automation gate

Refactor `scripts/release.sh` into prepare/build/verify/publish stages, with publication impossible until all gates pass:

1. Require a clean release branch, exact semantic display version, unique monotonic numeric build, pinned Sparkle resolution, release notes, and correct stable/beta channel.
2. Build UI, helper, server, DMG/package payload, and package scripts from the same commit. Do not commit or tag merely as a side effect before artifacts pass.
3. Sign Sparkle's nested code in documented order, sign apps with Developer ID Application, sign the flat package with Developer ID Installer, submit the package through `notarytool`, staple it, and assess it. Apple recommends `pkgutil --check-signature` for installer signatures and `stapler` for attaching notarization tickets. [Apple notarization issue resolution](https://developer.apple.com/documentation/security/resolving-common-notarization-issues), [Apple custom notarization workflow](https://developer.apple.com/documentation/security/customizing-the-notarization-workflow)
4. Verify `codesign --verify --deep --strict` on both final apps, `pkgutil --check-signature`, `spctl --assess --type install`, `xcrun stapler validate`, payload paths/ownership, framework symlinks, framework linkage/rpath, plist feed/key/version values, package version, and SHA-256.
5. Install the notarized package on a clean Apple Silicon macOS test user/host; verify first install, bridge install over the last legacy version, stable-to-stable Sparkle update, stable opt-in to beta, beta back to a higher stable build, authorization cancellation, bad signature/download failure, package-script failure fixture, helper relaunch, server health, user-data preservation, and manual recovery. Sparkle recommends testing from an old version and notes that production code-signing behavior may require a notarized build; package updates specifically may fail from ordinary development builds. [Sparkle test procedure](https://sparkle-project.org/documentation/), [Sparkle package updates](https://sparkle-project.org/documentation/package-updates/)
6. Stage versioned `.pkg` and release notes at their final HTTPS URLs; verify download size/hash; generate EdDSA enclosure metadata; validate XML/channel/build ordering; then atomically publish appcast last.
7. Only after the live appcast and package probe succeed should automation create/push the release tag and GitHub release. A prerelease must remain marked prerelease. Publishing must require an explicit confirmation flag/environment gate so ordinary CI can build and smoke-test without releasing.
8. Archive the exact package, appcast, release notes, EdDSA signature/length, notarization submission ID/log, checksums, build provenance, Sparkle version, and dSYMs. Sparkle recommends retaining its dSYMs for crash-symbolication. [Sparkle test/distribution guidance](https://sparkle-project.org/documentation/)

## Acceptance criteria

- Vox Helper uses pinned Sparkle 2 through a reproducible SwiftPM build and no Terminal window for normal updates.
- Its final notarized bundle contains a correctly linked, symlink-preserved, inside-out-signed `Sparkle.framework` and passes signature/Gatekeeper validation.
- A one-time bridge package upgrades the last legacy installation without losing user data.
- Stable users cannot see beta items; opted-in beta users can see beta and later default-channel releases.
- Every appcast enclosure references the immutable flat notarized package and has the exact EdDSA signature and byte length; build numbers are strictly increasing.
- Automatic checks may run, but package installation always presents authorization and cancellation is harmless.
- Download, signing, authorization, install, package-script, helper-relaunch, and server-health failures each leave an actionable log and a manual recovery path.
- The shell updater remains available and tested for recovery/source installs, while Sparkle is the sole standard update command.
- A notarized old-to-new production update and installer/update failure matrix passes on Apple Silicon before any appcast or release is published.
